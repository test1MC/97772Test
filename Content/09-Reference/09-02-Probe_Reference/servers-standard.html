Server Probes - Standard

InterMapper can test a number of standard servers.

Note: For each probe that compares a response string with a specified string, InterMapper checks to see if the specified string is contained in the response string. An exact match is not required.

  • Basic TCP - Connects to a TCP port
  • Basic TCP (Blocked) - Tests a blocked TCP port
  • Custom TCP - Connects to a TCP port, sends a string and checks the response
  • CVS Server - Tests a Concurrent Version System (CVS) server
  • DHCPv4/BOOTP - tests a DHCP server
  • Domain Name (DNS) - tests a DNS server
  • FTP Probes
    • FTP (Login) - Logs in and tests an FTP server
    • FTP (No login) - Tests an FTP server without logging in
  • Gopher- Tests a Gopher server
  • Host Resources - Monitors CPU, memory, and disk utilization
  • HTTP Probes
    • HTTP - Tests an HTTP server without SSL
    • HTTP Follow Redirects - Tests that a web server is redirecting certain URLs to a specific URL.
    • HTTP Post - Tests an HTTP server with a POST query without SSL
    • HTTP Proxy- Tests an HTTP Proxy server
    • HTTP Redirection- Test that HTTP redirection is working properly
  • HTTPS
    • HTTPS- Tests an HTTP server with SSL
    • HTTPS Post- Tests an HTTP server with a POST query with SSL
  • IPMI v2.0 - Uses Intelligent Platform Management Interface (IPMI) version 2.0 to test a Baseboard Management Controller (BMC).
  • IRC- Tests an IRC server
  • LDAP - Tests an LDAP server without SSL
  • LDAP-SSL- Tests an LDAP server with SSL
  • LPR - Tests an LPR print server
  • Mail Probes
    • IMAP4
      • IMAP4- Tests an IMAP4 server without SSL
      • IMAP4-SSL- Tests an IMAP4 server with SSL
    • POP3
      • POP3- Tests a POP3 server, without SSL
      • POP3-SSL - Tests a POP3 server, with SSL
    • SMTP
      • SMTP- Tests an SMTP server without encryption
      • SMTP-TLS- Tests an SMTP server with encryption
  • Multimedia
    • Multicast Listener - Tests a multicast stream
    • RTSP- Tests an RTSP streaming server
    • Session Initiation protocol (SIP) over UDP - Sends a UDP packet and tests for a valid SIP response.
  • Network Time (NTP) - Tests a Network Time Server
  • NNTP - Tests a Network News server
  • RADIUS - Tests a Radius authentication server
  • SNPP- Tests a SNPP server
  • SSH - Tests a remote login through a secure shell.
  • Subversion - tests a Subversion server, running either stand-alone, or as an Apache module.
  • Telnet- Tests a Telnet server
  • VNC Server- Tests a VNC server

Basic TCP

InterMapper attempts to establish a TCP connection to the specified port. The device is shown as UP if the connection completes within the default time interval (at which point the connection is disconnected); otherwise, the device is shown as DOWN.

Note: The default port value for the Basic TCP probe is 23, the Telnet port. You test many different kinds of servers by connecting to the proper port. For a list of port numbers and their common uses, see http://www.iss.net/security_center/advice/Exploits/Ports/.

Basic TCP (Blocked)

This probe tests to see if a TCP is not accepting connections. It can be used to make sure a firewall is working properly, or that a particular TCP service is never operating on an important machine. If the specified connection fails, device status is reported as OK. For each device, you can specify the reporting level to use if the port accepts the connection.

Custom TCP

InterMapper attempts to establish a TCP connection to the specified port. Once the connection has been established, InterMapper sends the "string to send" and waits the interval specified in the "seconds to wait" for a response to return from the device. The response text is compared to the specified OK, WARNING, ALARM, or DOWN strings, and the device is displayed in the corresponding state. If the response text does not match one of the specified strings, the device is considered to be DOWN.

CVS Server

InterMapper attempts to establish a TCP connection to port 2401 on a CVS server. It then attempts to log into the server using the supplied name and password. InterMapper scrambles the password as described in the CVS protocol specification, http://www.loria.fr/~molli/cvs/doc/cvsclient_4.html#SEC4.

DHCPv4/BOOTP

InterMapper sends a DHCP Discover or DHCP Inform request to the device as specified in the configuration window, and waits for an appropriate response.

InterMapper sends the request to the specified BootP Relay Address (normally the IP broadcast address, 255.255.255.255). If the BootP Relay Address is empty, InterMapper sends the request directly to the device. InterMapper then checks the response to ensure that the returned IP address is correct, and if specified, that the returned subnet mask and router address are correct. If it receives multiple responses, it gives a warning if the responses are inconsistent.

Domain Name (DNS) Probes

InterMapper sends the specified query to the DNS server being tested, and verifies that the response contains the correct information. Check the "Recursion Desired:" box to allow the target DNS server to interrogate other servers; leave it unchecked to verify that the target DNS server has a non-cached copy of the information. The device is set to the Failure Status if the response isn't correct.

  • Check the Recursion Desired to allow the target DNS server to interrogate other DNS servers recursively; leave it unchecked to verify that the target DNS server has a non-cached copy of the information. Force the DNS server being tested to use its own database, and return an error if the address is not available.

  • Choose a Failure Status dropdown menu to indicate the severity of a failure to receive a response or an incorrect answer in the response.

You can probe a DNS server using the following queries:

  • (A) Address - InterMapper sends a DNS "A" record query to the device for the specified Domain Name, and expects to receive the specified IP address in the DNS response.
  • (MX) Mail Server - Sends a request for a MX record for the specified Domain Name, and checks it against the DNS name specified in the Mail Server field.
  • (NS) Name Server - sends a request for a NS record for the specified Domain Name, and checks it against the DNS name specified in the Name Server field.
  • (PTR) Reverse Lookup - sends a request for a PTR record for the specified IP address, and checks it against the DNS name specified in the Domain Name field.
  • (TXT) Text Record - sends a request for a TXT record for the specified Domain Name, and checks it against the text string specified in the Text Substring field.

FTP Probes

FTP (Login)

InterMapper connects to a port (default is 21) on a File Transfer Protocol server (RFC 959) and attempts to log in using the specified username and password. It then issues a NOOP command, and finally a QUIT command.

FTP (No login)

InterMapper connects to a port (default is 21) on an FTP server (RFC 959). It then issues a NOOP command and a QUIT command. This command is useful for checking an FTP server frequently without filling its log files with login attempts.

Gopher

InterMapper connects to a port (default is 70) on a Gopher server (RFC 1436). It then sends the specified selector string (the default is an empty string). The presence of any response indicates that the server is up; no error checking is performed on the response.

HTTP Probes

HTTP

InterMapper connects to a port (default is 80) on an HTTP server (RFC 2068 & 1945). It then sends an HTTP GET request with the specified URL Path (the default is "/"). It then examines the resulting response for the presence of the "string to verify". You may also specify a name and password to be sent along with the HTTP request to test the authentication.

To test virtual hosting on a particular server, specify a DNS name in the "Host Name" parameter. InterMapper will include that in the HTTP Get request header.

HTTP (Follow Redirects)

This TCP probe lets you download a specific web page and scan it for a specific string of HTML. This probe will follow a limited number of page redirects to the same HTTP server.

HTTP (POST)

In order to test web form submissions, InterMapper can send an HTTP POST request to the server. InterMapper connects to a port (default is 80) and sends the specified URL Path and Form Data strings in the request. It then searches the response for the "string to verify".

HTTP (Proxy)

InterMapper can test proxy web servers to ensure that they are delivering pages properly. InterMapper sends an HTTP GET request containing a URL of some other host to the proxy server. If the response matches the "string to verify", the proxy server is shown as UP. You may optionally specify a proxy user ID and password which, by default, are empty.

HTTP (Redirect)

InterMapper uses the HTTP (Authenticate) probe to connect to the device on the specified port. It checks the response for the presence of a 301 Moved Permanently status giving the specified "Redirect path".

HTTPS Probes

HTTPS

InterMapper establishes an SSL connection to a HTTPS server (default port is 443), requests a page, and checks for the presence of a specified string on the resulting page. You can also specify the host name, the file's path, the expected string, and an optional user name and password. This probe is identical to the HTTP probe, except that it establishes the SSL connection.

HTTPS (Post)

InterMapper establishes a SSL connection to a HTTPS server (default port is 443) and posts form data to the server. This probe is identical to the HTTP (Post) probe, except that it establishes the SSL connection

Intelligent Platform Management Interface (IPMI v2.0)

This probe implements version 2.0 of the Intelligent Platform Management Interface (IPMI) over a LAN. It sends UDP-based RMCP+ packets to a Baseboard Management Controller (BMC) located within a server or workstation. The BMC is hardware which permits network-based management of the computer even when it is turned off, i.e. "lights-out management".

  • User is an administrator-level user name to the BMC. This parameter is required.
  • Password is the password for the desired administrator-level user. This parameter is required.
  • Dialect specifies the variant of the IPMI protocol. Unfortunately, there are subtle differences in implementations of IPMI in various products. To use this probe with an Apple XServe, set the "Dialect" parameter to "XServe". For Dell Servers and possibly other products, set the "Dialect" parameter to "Other".

This probe supports one-key, non-anonymous logins only. Internally, it uses RAKP-HMAC-SHA1 and AES-CBS-128 for authentication and confidentiality, respectively. The firewall configuration of the BMC must permit UDP packets from InterMapper.

IRC

InterMapper connects to a port (default is 6667) on an Internet Relay Chat (RFC 1459) server. It then logs in using the IRC protocol with the specified Nickname, Password, and Username. It then compares the response to the "string to verify".

LDAP

InterMapper connects to a port (the default is 389) on a Lightweight Directory Access Protocol (RFC 2251) server. It then attempts to use the specified "Bind Name" and optional "Bind Password" to look up the name specified in "Name to Lookup". If no response is returned, or if the response indicates an error, InterMapper shows the device as DOWN.

LDAP-SSL

InterMapper performs the same tests as the LDAP probe (above) over an SSL connection.

LPR

InterMapper connects to a port (default is 515) and requests information about a specified queue. If the response is correct, the device is considered to be UP.

Mail Probes

IMAP4 Probes

IMAP4

InterMapper connects to a port (the default is 143) on an Internet Message Access Protocol - Version 4 (RFC 2060) server. It then issues the IMAP4 CAPABILITY, NOOP, and LOGOUT commands sequentially. It also checks the response of the CAPABILITY command to ensure that the device supports IMAP4 or IMAP4rev1.

IMAP4-SSL

InterMapper performs same tests as the IMAP4 probe (above) over an SSL connection.

POP3 Probes

POP3

InterMapper connects to a port (default is 110) on a Post Office Protocol - version 3 (RFC 1939) server, and verifies that the response contains "+OK".

POP3-SSL

InterMapper connects to an SSL port (default is 995) on a Post Office Protocol - version 3 (RFC 1939) server, and verifies that the response contains "+OK".

SMTP Probes

SMTP

InterMapper connects to a port (default is 25) on a Simple Mail Transfer Protocol (RFC 821) server and attempts to verify an e-mail address. It connects, then sends a HELO command, a VRFY command with the specified "E-mail Address", and then a QUIT command.

SMTP-TLS

InterMapper performs the SMTP probe (above) over a connection using TLS encryption.

Multimedia Probes

Multicast Listener

InterMapper listens to the stream of packets arriving on the specified multicastaddress and UDP port.

  • If a packet fails to arrive within the specified time period, InterMapper sets the device status to DOWN.
  • If the Verify source address flag is true, then InterMapper accepts packets only from the device's IP address. Packets from other sources are ignored.

This is most useful for monitoring RTP streams, such as those sent by QuickTime and RealNetworks servers. The settings for this probe are contained in the Session Description Protocol (SDP) file associated with the multicast broadcast.

Open the SDP file using a text editor, and locate the required values into the configuration window. Here is an excerpt from an SDP file:

o=- 3151391188 3157116984 IN IP4 192.168.152.11 <--  source is 192.168.152.11
s=Sample QuickTime SDP File
i=Copyright @ 2005  
a=x-qt-text-nam: My QuickTime broadcast
a=x-qt-text-cpy:Copyright © My Broadcast
a=x-qt-text-cmt:Broadcast by Sorenson Broadcaster 1.00.307.100
c=IN IP4 224.2.166.240/15/1    <-- multicast address is 224.2.166.240
t=3157116984 0
m=audio 18562 RTP/AVP 12 <-- port for the audio is 18562
a=rtpmap:99 X-QT/600
a=rtpmap:102 X-QDM
a=x-bufferdelay:8

Note: An SDP file may describe a broadcast with many tracks (audio, video, text, etc.) Each of these has a "m= xxxxx port#" line. InterMapper may monitor multiple tracks, but uses a separate device for each.

RTSP

InterMapper connects to a port (default is 554) to on a Real Time Streaming Protocol server (RFC's 1889 & 2326) and verifies its operation. It causes the specified movie to play for the specified number of seconds by sending a sequence of CONNECT, START, and STOP commands.

Session Initiation Protocol (SIP) over UDP

The protocol used to set up voice communications for Voice over IP (VOIP), as described in RFC 3261. This probe sends a SIP request in a single UDP packet and checks for a valid SIP response.

By default, this probe sends an OPTIONS command to the target device. However, some VOIP systems will not answer un-authenticated OPTIONS requests. For these devices, change the command to REGISTER.

  • URI is the SIP uniform resource identifier in the request.
  • Command is the SIP command to send in the request.

Network Time Server

This probe sends a client-mode request to an NTP server asking for the current time, and displays an error if no response is returned. By default, NTP requests are sent on UDP port 123.

Note: This probe does not validate the time in the NTP response.

NNTP

InterMapper connects to a port (default is 119) on an Network News Transfer Protocol (RFC 977) server and requests information about the specified group with a GROUP command. It then sends a QUIT command to terminate the session.

RADIUS (Remote Authentication Dial-in User Service)

This probe tests a RADIUS server by sending an Access-Request packet to authenticate a specific user name and password. (A RADIUS server does not answer access-requests from a client it doesn't recognize.)

Before you can use this probe with a particular RADIUS server, you must add the InterMapper computer's IP address to the RADIUS server and choose a "shared secret" for it. The "shared secret" is used by the RADIUS protocol to encrypt passwords in RADIUS requests.

Simple Network Paging Protocol (SNPP)

This probe tests a SNPP server using the protocol defined in RFC 1861. It connects to the server (default port is 444), and then issues a PAGE pagerid command. If it receives a valid response, it issues a QUIT command to the server and sets the device status to OK. If an "Invalid pager ID" response comes back, InterMapper marks the device in the Alarm state. If no connection could be made, or if the response is unknown, the device is set to the DOWN status.

Secure Shell (SSH)

This TCP probe opens a connection to the specified port and looks for the identification string that indicates an SSH server as specified in FRC 4253. The default TCP port number for SSH connections is port 22.

Subversion

Subversion Server (Apache module)

This probe tests a Subversion server running as an Apache module. The subversion module lets Apache function as a WebDAV/DeltaV server. Since the server responds normally to HTTP GET requests, testing whether it is up is the same as performing an HTTP GET request and checking to ensure the location was found.

Subversion Standalone Server (svnserve)

This probe tests a stand-alone svnserve Subversion server. It connects to the svnserve using its default port 3690. The Sserver returns a response to indicate it is running. If a repository location is specified, the probe then tries to connect to that repository. If a username is specified, the probe tries to authenticate using CRAM-MD5, otherwise it connects anonymously.

Telnet

InterMapper connects to a port (default is 23) on a Telnet (RFC 845) server and attempts log in using the specified name and password. You also have the option to enter a command.

VNC Server

This probe tests a VNC (Virtual Network Computer) server by connecting to port 5900 (by default). If it receives "RFB.003.003" response, the server is assumed to be UP. If no connection is made, or if the response is unknown, then the device is considered DOWN. VNC is described at: http://www.realvnc.com
